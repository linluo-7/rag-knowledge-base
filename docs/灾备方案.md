# 灾备方案文档

> RAG 知识库系统灾难恢复计划

## 1. 恢复目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| RPO | 1小时 | 数据丢失最多 1 小时 |
| RTO | 4小时 | 服务恢复最多 4 小时 |
| 可用性 | 99.9% | 全年不可用 < 9 小时 |

---

## 2. 备份策略

### 2.1 Milvus 备份

```bash
# 备份 Collection
milvus-backup create --name daily_backup

# 列出备份
milvus-backup list

# 恢复
milvus-backup restore --name daily_backup
```

**备份频率**:
| 数据类型 | 频率 | 保存时间 |
|---------|------|---------|
| 全量 | 每天 | 30 天 |
| 增量 | 每小时 | 7 天 |
| 事务 | 实时 | Kafka |

### 2.2 Neo4j 备份

```bash
# 导出
neo4j-admin database dump neo4j --to-path=/backups

# 导入
neo4j-admin database load neo4j --from-path=/backups
```

### 2.3 配置备份

```yaml
# 备份脚本 (cron)
0 2 * * * /app/scripts/backup.sh
```

## 3. 故障场景

### 3.1 Milvus 服务故障

**现象**: API 503 向量检索失败

**恢复步骤**:
```bash
# 1. 检查状态
docker compose ps

# 2. 重启服务
docker compose restart milvus

# 3. 如果还是不行，从备份恢复
milvus-backup restore --name daily_backup

# 4. 验证
curl http://localhost:19530/healthz
```

**恢复时间**: < 30 分钟

---

### 3.2 Neo4j 图数据库故障

**现象**: 图谱查询失败

**恢复步骤**:
```bash
# 1. 重启
docker compose restart neo4j

# 2. 从备份恢复（如果数据损坏）
docker compose exec neo4j neo4j-admin database load neo4j --from-path=/backups
```

**恢复时间**: < 1 小时

---

### 3.3 整个服务不可用

**现象**: 所有 API 失败

**恢复步骤**:
```bash
# 1. 切换到备份环境
docker compose -f docker-compose.backup.yml up -d

# 2. 从备份恢复数据
/app/scripts/restore.sh

# 3. DNS 切换
curl -X POST http://dnslb/switch
```

**恢复时间**: < 4 小时

## 5. 监控告警

### 5.1 关键指标

| 指标 |阈值 | 动作 |
|-------|-----|------|
| API 错误率 | > 5% | 告警 + 切换 |
| 向量检索延迟 | > 2s | 告警 |
| 数据库连接 | 失败 | 立即告警 |
| 磁盘使用率 | > 80% | 告警 |

### 5.2 告警通知

```yaml
# Alertmanager 配置
- name: rag-critical
  receiver: on-call-phone
  repeat: 5m
```
